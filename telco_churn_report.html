01

Incident log: data quality

Three issues surfaced during intake. Each got an investigated, documented decision rather than a silent fix — the kind of paper trail a retention system needs before it's trusted with real customer data.

INC-001 · TotalCharges RESOLVED

Column loads as text, not currency

Eleven rows contained blank strings, which silently typed the entire column as text. Investigation traced all eleven to customers with tenure = 0 — brand new accounts that have not yet been billed.

Fix appliedThese are not missing values — the true amount charged is zero. Imputed 0, not the median, and kept the rows: new customers are exactly the population a churn model must score.
INC-002 · SeniorCitizen RESOLVED

Inconsistent encoding against every other binary field

Stored as 0/1 while every other yes/no column in the dataset used the strings "Yes"/"No" directly.

Fix appliedRemapped to Yes/No so one encoder handles all categoricals, and plots read consistently without a lookup table.
INC-003 · Class balance RESOLVED

26.5% churn rate makes accuracy a misleading target

A model predicting "no churn" for every customer scores 73.5% accuracy while catching zero at-risk customers.

Fix appliedStratified train/test split, class-weighted logistic regression, and evaluation led by ROC-AUC, PR-AUC, and recall on churners — never accuracy alone.
02

What actually drives churn

Four signals stood out clearly enough in the exploratory analysis to shape both the model and the retention playbook.

~10×
Contract type

Month-to-month customers churn at roughly ten times the rate of two-year contracts — the single strongest signal, and the most actionable: converting a high-risk customer to an annual plan directly attacks the risk factor.

Yr 1
Tenure

Churn is heavily front-loaded in the first twelve months. Retention effort has the highest expected value in the onboarding window, not spread evenly across the base.

Fiber
Internet service

Fiber optic customers churn far more than DSL despite being the premium tier — a price and expectation problem more than a product one.

~2×
Payment method

Electronic check payers churn at about double the rate of autopay customers. Manual monthly payment is a monthly decision point to cancel; autopay removes that friction.

Model comparison

Two candidates trained on an 80/20 stratified split, tracked in MLflow. The metric that matters here is recall on churners, not accuracy.

The boosted model posts higher accuracy, which is exactly why accuracy is the wrong headline metric for this problem: it buys that number by missing nearly half of all churners. The class-weighted logistic regression catches 78% of churners and ships with coefficients the retention team can actually interpret — selected for both reasons.

What's next

  • Threshold tuning against an explicit cost matrix — retention offer cost vs. customer lifetime value — instead of a flat 0.5 cutoff.
  • SHAP-based per-prediction explanations returned alongside each score, so the retention team sees why, not just the number.
  • A great-expectations checkpoint gating retraining, so a future data drift fails loudly instead of silently degrading the model.
  • GitHub Actions: tests → image build → push → App Runner auto-deploy on every merge to main.
